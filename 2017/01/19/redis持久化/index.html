redis持久化

发表于 2017-01-20 | 分类于 redis
让数据更安全....

简介

redis是一个支持持久化的内存数据库，也就是说其将数据存储在内存中，因此速度相当快，但是我们知道内存中的数据一旦断电或者重启那么数据将全部丢失，不用担心，redis支持持久化的功能，也就是说将内存中的数据写入磁盘中，一旦断电或其他故障，可以将磁盘中的数据再次写入内存中。redis支持两种持久化，一种是基于快照方式的RDB，一种是基于类似MySQL的二进制日志的AOF。

RDB

RDB就是Snapshot快照存储，是默认的持久化方式，通过配置文件中save参数配置的周期将数据自动保存在磁盘上，也可以人为通过save或者bgsave命令来做一次快照持久化，对应产生的数据文件为dump.rdb。

save命令

save操作是在主进程中保存快照的，由于redis是用一个主线程来处理所有 client的请求，这种方式会阻塞所有client请求，因此不推荐使用。

bgsave命令

bgsave操作是主进程fork出一个子进程，由子进程来负责快照持久化，因此主进程还是可以处理client请求。

问题

每次快照持久化都是将内存数据完整写入到磁盘一次,并不是增量的只同步脏数据。如果数据量大的话,而且写操作比较多,必然会引起大量的磁盘io操作,可能会严重影响性能。

优缺点

优点

1.RDB在重启保存了大量数据实例的时候比AOF快。

2.RDB在保存了大量数据实例的时候生成的文件比AOF文件通常要小。

缺点

1.在一些事故(比如断电)的情况下，RDB会丢失一个周期的数据

AOF

AOF比RDB有更好的持久化性,由于在使用AOF持久化方式时,Redis会将每一个收到的写命令都通过Write函数追加到文件最后,类似于MySQL的binlog，当Redis重启是会通过重新执行文件中保存的写命令来在内存中重建整个数据库的内容。默认情况下AOF是关闭的，打开需要修改配置文件的appendonly no为appendonly yes，生成的文件名字默认为appendonly.aof。

优点

1.比RDB有更好的持久性

2.AOF日志是一个追加文件，所以不需要定位，在断电时也没有损坏问题。即使由于某种原因文件末尾是一个写到一半的命令(磁盘满或者其他原因),redis-check-aof工具也可以很轻易的修复。

缺点

1.对同样的数据集，AOF文件通常要大于等价的RDB文件。

2.AOF可能比RDB慢，这取决于准确的fsync策略。通常fsync设置为每秒一次的话性能仍然很高，如果关闭fsync，即使在很高的负载下也和RDB一样的快。不过，即使在很大的写负载情况下，RDB还是能提供能好的最大延迟保证。

AOF文件过大

为了合并重写AOF的持久化文件，Redis提供了BGREWRITEAOF命令。收到此命令后Redis将使用与快照类似的方式将内存中的数据以命令的方式保存到临时文件中，最后替换原来的文件，以此来实现控制AOF文件的合并重写。由于是模拟快照的过程，因此在重写AOF文件时并没有读取旧的AOF文件，而是将整个内存中的数据库内容用命令的方式重写了一个新的AOF文件。(至于文件为什么会变小，其实很简单，比如一个队列，append一个数据，pop一个数据，这样反复几次，其实相当于没有变化，但是AOF会全部以命令的方式记录下面，而BGREWRITEAOF会去掉这些没有实际意思的命令，文件当然会变小。)

优先级

在AOF与RDB同时开启的时候，redis重启优先使用AOF文件来恢复数据。

恢复策略

1.如果能允许少量的数据丢失，建议关闭AOF然后启动redis，这样就使用RDB文件来恢复数据，速度会快很多。

2.如果不允许数据丢失，那么建议先使用BGREWRITEAOF命令来重写AOF文件，去掉一些冗余的命令，然后开启AOF，重启redis，这样数据不会丢失，但是速度会慢很多。
