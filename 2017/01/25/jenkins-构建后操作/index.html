jenkins 构建后操作

发表于 2017-01-26 | 分类于 jenkins
深入理解项目构建之后的可操作动作.

Aggregate downstream test result

因为在测试中执行时间通常是占主要地位的,一个好的jenkins实践包含了将测试执行分开到不同的项目中
去,可能是多重不同的项目.
当这样做了，设置测试聚会对于从下游测试项目中收集所有的测试结果并且在构建的时候展示出来是一个
十分方便的方式,通过这种方式可以更快更方便的看到所有被构建的测试的状态.

Archive the artifacts

生成归档文件(例如,zip和jar格式)是为了以后能够下载,归档文件可以在jenkins的网页上访问.
通常如果一个构建日志被保存下来了jenkins就会保存归档文件.但是如果不需要旧的归档文件并且更倾向
于节省磁盘空间的话,你可以这样做.

Build other projects

构建其他的项目

Deploy artifacts to Maven repository

将项目的artifacts部署到一个Maven仓库,与标准的mvn deploy比较起来,这个特性允许你在整个的构建
被确认成功的时候部署artifacts.

Publish Performance test result report

这个插件能够理解JMeter分析报告的XML格式,JMeter Summariser报告文本格式, SOAPUI JUnit格式
报告,和Iago格式.
此插件不执行实际分析,只显示有关分析结果的有用信息,例如，平均响应时间,历史结果趋势,查看分析报
告的WEB用户界面等.
要使用此功能,首先建立运行测试，然后选择解析器(JMeter, JUnit, or Iago),最后必须指定不同性能
文件的路径,默认情况下,对于JMeter使用**/*.jtl,对于Junit测试使用**/TEST*.xml,对于JMeter 
Summariser使用**/*.log,对于Iago使用parrot-server-stats.log.

Record fingerprints of files to track usage

jenkins能够记录文件的指纹(大多数情况下是jar文件)以跟踪这些文件什么时候产生的和被使用的,当在
jenkins上有内部依赖的项目,能快速找出问题的解决方法,例如:
    1.在我的磁盘上有foo.jar文件,但是这个文件是从哪里产生的?
    2.BAR项目依赖于从FOO项目产生的foo.jar文件
        构建生成foo.jar的什么东西会被BAR项目需要?
        构建生成的BAR项目对于foo.jar包含bug修复吗?
要想使用这个功能,所有相关的项目(不仅仅是生成的文件的项目,而且包含被使用的文件的项目)都需要使
用它来记录指纹信息

Git Publisher

可选性的将结果,标记,分支推送到远程版本库中.

Deploy war/ear to a container

将war包或者ear包部署到一个容器中.

Trigger parameterized build on other projects

这个插件触发器是构建在其它的项目上的,其参数是预定义或者由结束的构建来提供.
即使目标没有参数化,又或者没有定义该名字的属性,每一个参数都会传递到项目中.
